# Предварительный анализ архитектуры сайта

Статус: `draft-for-architecture-decision`

## 1. Как я понимаю задачу

Нужно подготовить техническую основу для публичного сайта `Core AI Platform`.
Сейчас задача не в том, чтобы верстать финальный контент, а в том, чтобы
заранее выбрать стек, структуру директорий и правила разработки для серьезного
многоязычного сайта.

Сайт должен:

- поддерживать `English`, `Spanish`, `Russian`;
- легко расширяться новыми языками;
- разворачиваться на обычном хостинге без сложного DevOps;
- использовать контентную основу из `content/`;
- иметь понятную SEO-структуру;
- отделять движок, шаблоны, локализации, ассеты и документацию;
- быть удобным для дальнейшей поддержки другим разработчиком.

В качестве архитектурного ориентира используется готовый PHP-сайт:

```text
/Volumes/Work/DV/ web_kassa/site
```

## 2. Рекомендация по стеку

Рекомендуемый стек для первой версии:

```text
PHP 8.4 preferred
PHP 8.3 minimum for production
+ HTML templates
+ CSS
+ vanilla JavaScript
+ file-based localization
+ no frontend framework
+ no database for v1
```

Причина: сайт должен легко размещаться у хостера. PHP остается самым простым и
практичным вариантом для обычного shared/VPS-хостинга: достаточно загрузить
файлы, настроить домен, PHP-версию и права на служебные директории.

`PHP 8.2` можно рассматривать только как временный компромисс, если конкретный
хостер еще не дает `8.3/8.4`. Для нового production-сайта нельзя начинать с
версии PHP, которая уже вышла из поддержки или близка к ее окончанию.

## 3. Почему не React / Next / Node для v1

React, Next.js или Node.js можно использовать, если сайт превращается в сложное
web-приложение. Для текущей задачи это преждевременно.

Минусы такого подхода для v1:

- сложнее хостинг;
- нужен build/deploy pipeline;
- больше зависимостей;
- выше риск сломать SEO из-за неправильного рендера;
- сложнее поддержка для простого маркетингового сайта;
- часть хостеров не поддерживает Node.js без отдельной настройки.

Для текущей цели PHP дает лучший баланс: простое развертывание, серверный
рендеринг, нормальное SEO, понятная мультиязычность и минимум инфраструктуры.

## 4. Архитектурный принцип

Главный принцип:

```text
Один движок и шаблоны, разные языковые словари.
```

Нельзя делать три независимые копии сайта под каждый язык. Это быстро приведет
к расхождению контента, метаданных, меню, ссылок и SEO.

Правильная модель:

- единая точка входа;
- единые шаблоны страниц;
- отдельные файлы локализации;
- единый helper для переводов;
- fallback на базовый язык;
- централизованные SEO-метаданные;
- единые правила генерации URL.

## 5. Рекомендуемая URL-модель

Для сайта v1 рекомендуется такая схема:

```text
/        auto language redirect
/en/     English, fallback language
/ru/     Russian
/es/     Spanish
```

Язык выбирается по явному URL, cookie ручного выбора или заголовку браузера
`Accept-Language`. Если подходящий язык не найден, используется английский.

Для будущих страниц:

```text
/en/platform/
/en/agents/
/en/pilot/

/ru/platform/
/ru/agents/
/ru/pilot/

/es/platform/
/es/agents/
/es/pilot/
```

## 6. Рекомендуемая структура директорий

Целевая структура проекта:

```text
site/
  index.php
  robots.txt
  sitemap.xml

  config/
    app.php

  includes/
    bootstrap.php
    helpers.php
    i18n.php
    seo.php
    routing.php

  locales/
    ru.php
    en.php
    es.php

  templates/
    layout.php
    pages/
      home.php
      platform.php
      agents.php
      pilot.php
    partials/
      head.php
      header.php
      footer.php
      language-switcher.php

  handlers/
    contact.php

  assets/
    images/
    icons/
    logos/
    social/

  static/
    styles.css
    script.js

  storage/
    logs/

  content/
    ...

  docs/
    ...

  TZ/
    ...
```

Если хостинг не позволяет скрывать внутренние папки от web-доступа, нужно
добавить `.htaccess`-правила или перенести публичную часть в `public/`. Для
обычного простого хостинга можно стартовать с `index.php` в корне, как в
референсном PHP-сайте, но доступ к внутренним папкам должен быть закрыт
серверными правилами.

## 7. Назначение ключевых частей

`config/` хранит настройки сайта: домен, базовый язык, список языков, имя
проекта, email, SEO defaults, пути к social image.

`includes/` хранит служебную PHP-логику: загрузка конфига, выбор локали,
безопасный вывод, генерация URL, SEO helpers, fallback-переводы.

`locales/` хранит все переводимые данные: тексты страниц, меню, кнопки, alt,
meta title, meta description, Open Graph-тексты, сообщения форм.

`templates/` хранит HTML/PHP-шаблоны. Шаблоны не должны содержать большие
текстовые блоки напрямую. Они только выводят данные из локали.

`handlers/` хранит серверные обработчики: форма заявки, будущие AJAX-endpoints,
anti-spam, валидация.

`assets/` хранит изображения и визуальные ресурсы.

`static/` хранит CSS и JS. Если проект останется совсем маленьким, можно
держать `styles.css` и `script.js` в корне, но отдельная папка чище.

`storage/` хранит runtime-файлы: логи, временные записи, rate-limit файлы.
Эта директория не должна попадать в публичную индексацию.

`content/` остается редакционной зоной и source of truth для смыслов сайта.

`docs/` и `TZ/` хранят проектную документацию, правила и технические задания.

## 8. SEO-основа

SEO должно быть частью движка, а не финальной правкой перед релизом.

В v1 нужно заложить:

- уникальные `title` и `meta description` для каждой страницы и языка;
- корректный `<html lang="">`;
- `canonical`;
- `hreflang` для `ru`, `en`, `es` и `x-default`;
- self-referencing `hreflang` на каждой языковой версии;
- полностью квалифицированные абсолютные URL в `canonical` и `hreflang`;
- Open Graph и Twitter Card;
- локализованные alt-тексты;
- `robots.txt`;
- `sitemap.xml` со всеми языковыми URL;
- отсутствие конфликтов между `robots.txt`, `noindex`, `canonical` и sitemap;
- JSON-LD structured data для `Organization`, `WebSite`, `WebPage`;
- человеко-понятные URL;
- отсутствие дублей между языковыми страницами.

`meta keywords` не должен считаться обязательным SEO-инструментом. Для Google
он не является полезным сигналом ранжирования; важнее нормальные заголовки,
описания, структура контента, внутренние ссылки, индексируемые страницы и
качественный видимый текст.

## 9. Требования к производительности

Для современного сайта v1 нужно учитывать:

- минимальное число CSS/JS-файлов;
- оптимизированные изображения;
- `webp/avif`-версии ключевых изображений, если это не ломает поддержку;
- responsive images через `srcset`/`sizes` для крупных иллюстраций;
- отдельные social preview изображения;
- lazy loading для неключевых изображений;
- нельзя ставить `loading="lazy"` на LCP-изображение первого экрана;
- для LCP-изображения можно использовать `fetchpriority="high"`;
- отсутствие тяжелого JS без необходимости;
- серверный HTML, доступный без JavaScript;
- фиксированные размеры или aspect-ratio для изображений, чтобы не было layout shift;
- gzip/brotli-сжатие на хостинге;
- долгий cache-control для versioned assets;
- короткий cache-control для HTML;
- проверка мобильной версии как основной пользовательской поверхности.

Целевые показатели Core Web Vitals:

- `LCP <= 2.5s`;
- `INP <= 200ms`;
- `CLS <= 0.1`;
- оценивать нужно 75-й перцентиль отдельно для mobile и desktop.

## 10. Требования к доступности

Минимальная база:

- семантические теги `header`, `main`, `section`, `footer`, `nav`;
- один главный `h1` на страницу;
- логичная иерархия заголовков;
- alt-тексты для meaningful images;
- пустой `alt=""` для декоративных изображений;
- aria-label только там, где это действительно нужно;
- фокус-состояния для кнопок и ссылок;
- форма с label, ошибками и понятными сообщениями.

## 11. Требования к безопасности

Даже маркетинговый сайт должен быть собран безопасно:

- весь вывод пользовательских/контентных строк экранировать;
- не хранить SMTP-пароли в публичном репозитории;
- добавить `config/*.example.php` для примеров;
- валидировать формы на сервере;
- добавить CSRF-токен для формы;
- ограничить частоту отправки формы;
- не показывать технические ошибки пользователю;
- логировать ошибки в `storage/logs`;
- закрыть доступ к `config`, `includes`, `storage`, `content`, `docs`, `TZ`.

Также нужно заложить базовые HTTP security headers:

- `Content-Security-Policy`;
- `X-Content-Type-Options: nosniff`;
- `Referrer-Policy: strict-origin-when-cross-origin`;
- `Permissions-Policy` с минимально нужными разрешениями;
- `Strict-Transport-Security` после включения стабильного HTTPS;
- корректные `Content-Type` и `charset=utf-8`.

Для CSP лучше начать с умеренной политики и отдельного режима проверки, если
на сайте будут сторонние виджеты, аналитика или CDN.

## 12. Требования к браузерам и совместимости

Сайт должен корректно работать в актуальных версиях:

- Chrome;
- Safari;
- Firefox;
- Edge;
- мобильных браузеров iOS и Android.

Минимальные требования:

- валидный HTML5;
- UTF-8 на всех страницах;
- `<meta name="viewport">`;
- отсутствие критичного контента, доступного только через JavaScript;
- корректные favicon/app icons;
- graceful degradation для декоративных эффектов;
- отсутствие console errors в production;
- проверка форм, меню и language switcher на touch-устройствах.

## 13. Предварительное решение

На текущем этапе оптимальное решение:

```text
Делать сайт на современном PHP без фреймворка,
использовать референс web_kassa как архитектурный образец,
но собрать более компактную версию под ru/en/es,
с централизованной локализацией, SEO helpers и строгими правилами разработки.
```

Такой подход даст простой деплой у хостера и не закроет путь к расширению:
позже можно добавить новые языки, новые страницы, legal-раздел, форму заявки,
блог или документацию без переписывания основы.
