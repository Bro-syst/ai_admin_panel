# Универсальный архитектурный blueprint для многоязычного PHP-сайта

Статус: `reusable-site-standard`

Дата: `2026-06-15`

## 1. Назначение документа

Этот документ описывает повторяемый подход к созданию современных
многоязычных сайтов на PHP. Его можно использовать как стартовый стандарт для
нового проекта, чтобы разработчик сразу понимал:

- какой стек использовать;
- как строить структуру директорий;
- как проектировать локализацию;
- как организовать SEO;
- как разделять контент, шаблоны и код;
- какие правила безопасности и качества обязательны;
- как готовить сайт к простому размещению у хостера.

Документ не привязан к конкретному бренду или текущему сайту. Это общий каркас,
который можно копировать в новый проект и адаптировать под его языки, страницы,
бренд, домен и бизнес-задачу.

## 2. Когда использовать этот подход

Подход подходит для:

- публичных B2B-сайтов;
- промо-сайтов продуктов;
- многоязычных корпоративных сайтов;
- сайтов услуг;
- лендингов с несколькими supporting pages;
- сайтов, которые должны легко разворачиваться на обычном PHP-хостинге.

Подход особенно полезен, если сайт должен быть:

- серьезным по восприятию;
- SEO-friendly;
- быстрым;
- безопасным;
- простым в сопровождении;
- расширяемым по языкам и страницам.

## 3. Базовое технологическое решение

Рекомендуемый стек:

```text
PHP 8.4 preferred
PHP 8.3 minimum for production
HTML templates
CSS
vanilla JavaScript
file-based localization
no frontend framework for v1
no database for v1 unless required
```

Если хостер поддерживает более новую стабильную версию PHP, ее можно
использовать после проверки совместимости. Если хостер дает только `PHP 8.2`,
это допустимо только как временный компромисс, но не как целевой стандарт для
нового production-сайта.

## 4. Почему PHP без фреймворка

Для сайта такого типа PHP без тяжелого фреймворка дает хороший баланс:

- простое размещение у хостера;
- серверный HTML сразу доступен поисковикам;
- нет обязательного build pipeline;
- меньше зависимостей;
- проще переносить сайт между серверами;
- проще объяснить архитектуру другому разработчику;
- легче контролировать SEO, sitemap, `hreflang`, canonical и HTML.

Фреймворк или Node.js оправданы, если сайт становится полноценным приложением:
личные кабинеты, сложные интерактивные сценарии, API, большое количество
динамических данных, админка, база данных. Для первого публичного сайта это
обычно лишняя сложность.

## 5. Главный архитектурный принцип

Главный принцип:

```text
Один движок, один набор шаблонов, разные словари локализации.
```

Нельзя делать отдельную копию сайта под каждый язык. Это приводит к расхождению
страниц, SEO, меню, ссылок, форм, footer и визуального поведения.

Правильная модель:

- одна система роутинга;
- один набор шаблонов;
- отдельные словари для языков;
- единый fallback-язык;
- единый механизм SEO;
- единый language switcher;
- единые правила генерации URL;
- единая форма и обработчик формы.

## 6. Рекомендуемая структура директорий

Базовая структура:

```text
site/
  index.php
  router.php
  robots.txt
  sitemap.xml

  config/
    app.php
    mail.example.php

  includes/
    bootstrap.php
    helpers.php
    i18n.php
    routing.php
    seo.php
    forms.php

  locales/
    en.php
    ru.php
    es.php

  templates/
    layout.php
    pages/
      home.php
      platform.php
      services.php
      contact.php
    partials/
      head.php
      header.php
      footer.php
      language-switcher.php
      contact-form.php

  handlers/
    contact.php

  assets/
    images/
    icons/
    logos/
    social/

  static/
    styles.css
    script.js

  storage/
    logs/
    cache/

  content/
    README.md
    source-copy/

  docs/
    00_PREDEVELOPMENT_ANALYSIS_RU.md
    01_DEVELOPMENT_RULES_RU.md
    02_LOCAL_RUN_RU.md
    03_DEVOPS_DEPLOY_RU.md

  TZ/
    01_SITE_ENGINE_ARCHITECTURE_TZ_RU.md
```

Если хостинг позволяет настроить document root на `public/`, можно использовать
более строгую модель:

```text
site/
  public/
    index.php
    robots.txt
    sitemap.xml
    assets/
    static/

  app/
    config/
    includes/
    locales/
    templates/
    handlers/

  storage/
  content/
  docs/
  TZ/
```

Для обычного shared-хостинга проще начинать с `index.php` в корне, но тогда
нужно закрыть внешний доступ к внутренним папкам серверными правилами.

## 7. Назначение основных директорий

`config/` хранит настройки сайта: бренд, домен, список языков, fallback-язык,
URL-структуру, email, SMTP, social image, runtime-параметры.

`includes/` хранит служебную PHP-логику: bootstrap, routing, i18n, SEO,
экранирование, формы, helpers.

`locales/` хранит переводимые данные: тексты страниц, меню, CTA, формы,
сообщения, alt-тексты, meta title/description, Open Graph.

`templates/` хранит HTML/PHP-шаблоны. Шаблоны отвечают за структуру, а не за
хранение больших текстов.

`handlers/` хранит серверные обработчики: контактные формы, заявки, будущие
AJAX-сценарии.

`assets/` хранит изображения, логотипы, иконки, social preview.

`static/` хранит CSS и JavaScript.

`storage/` хранит runtime-данные: логи, cache, временные файлы, rate-limit
записи. Эта папка не должна быть публичной.

`content/` хранит редакционные исходники: черновики, master-copy, sitemap,
позиционирование, тексты. Runtime не должен читать эти файлы напрямую.

`docs/` и `TZ/` хранят инженерные решения, правила и технические задания.

## 8. Конфигурационный контракт

В `config/app.php` должен быть единый контракт настроек:

```php
<?php

declare(strict_types=1);

return [
    'brand' => 'Example Brand',
    'base_url' => 'https://example.com',
    'default_locale' => 'en',
    'locales' => [
        'en' => ['label' => 'EN', 'name' => 'English', 'prefix' => 'en'],
        'ru' => ['label' => 'RU', 'name' => 'Русский', 'prefix' => 'ru'],
        'es' => ['label' => 'ES', 'name' => 'Español', 'prefix' => 'es'],
    ],
    'locale_cookie' => [
        'name' => 'example_locale',
        'max_age' => 31536000,
    ],
    'pages' => [
        'home' => '',
        'platform' => 'platform',
        'services' => 'services',
        'contact' => 'contact',
    ],
    'social_image' => '/assets/social/og-default.png',
];
```

Правила:

- конфиг должен быть единственным местом, где перечислены языки и страницы;
- секреты не хранить напрямую в `config/app.php`;
- для секретов использовать переменные окружения или закрытый файл в
  `storage/secrets.php`;
- все URL должны строиться через helper, а не руками в шаблонах.

## 9. Модель локализации

Рекомендуемая модель:

- все публичные языковые версии имеют явный префикс;
- корень `/` используется только для автоопределения языка и редиректа;
- fallback-язык задается в конфиге;
- язык явного URL имеет приоритет над cookie и браузером;
- ручной выбор языка сохраняется в cookie;
- отсутствующие ключи берутся из fallback-локали.

Пример URL:

```text
/        auto language redirect
/en/     English home
/ru/     Russian home
/es/     Spanish home

/en/services/
/ru/services/
/es/services/
```

Алгоритм выбора языка:

1. Если URL содержит явный языковой префикс, использовать его.
2. Если пользователь пришел на `/`, проверить cookie ручного выбора.
3. Если cookie нет, разобрать `Accept-Language` с учетом `q`.
4. Если подходящего языка нет, использовать fallback-язык.
5. Выполнить временный `302`-редирект на явный языковой URL.

Важно: `/ru/services/` всегда должен оставаться русской страницей. Нельзя
перебивать явный URL автоопределением браузера.

## 10. Структура словаря локализации

Каждый файл в `locales/` должен возвращать массив одинаковой структуры.

Пример:

```php
<?php

declare(strict_types=1);

return [
    'locale' => [
        'code' => 'en',
        'html_lang' => 'en',
        'label' => 'English',
    ],
    'meta' => [
        'home' => [
            'title' => 'Example Brand - Business Platform',
            'description' => 'Short localized description for search results.',
        ],
    ],
    'nav' => [
        'items' => [
            ['page' => 'home', 'label' => 'Home'],
            ['page' => 'services', 'label' => 'Services'],
            ['page' => 'contact', 'label' => 'Contact'],
        ],
    ],
    'home' => [
        'hero' => [
            'eyebrow' => 'Business platform',
            'title' => 'Build serious digital operations',
            'lead' => 'Short product explanation.',
            'primary_cta' => 'Request demo',
        ],
    ],
    'forms' => [
        'contact' => [
            'name_label' => 'Name',
            'email_label' => 'Email',
            'submit' => 'Send request',
            'success' => 'Thank you. We will contact you soon.',
            'error' => 'Please check the fields and try again.',
        ],
    ],
];
```

Правила:

- структура ключей во всех языках одинаковая;
- переводятся не только тексты секций, но и `alt`, формы, ошибки, CTA, SEO;
- шаблоны не должны содержать видимый текст напрямую;
- длинные строки на английском/испанском/немецком и других языках проверять в UI;
- для RTL-языков заранее предусмотреть `dir="rtl"` и отдельные проверки.

## 11. Шаблоны и страницы

Шаблоны должны быть простыми и предсказуемыми:

- `templates/layout.php` - общий каркас страницы;
- `templates/partials/head.php` - meta, canonical, hreflang, OG, JSON-LD;
- `templates/partials/header.php` - логотип, навигация, CTA;
- `templates/partials/footer.php` - footer, legal links, контакты;
- `templates/partials/language-switcher.php` - переключатель языков;
- `templates/pages/*.php` - содержимое конкретных страниц.

Правила:

- шаблон получает уже выбранную локаль и переводы;
- бизнес-логика не должна жить в HTML;
- в шаблоне можно использовать циклы по массивам из локали;
- сложные вычисления выносить в `includes/`;
- повторяемые блоки выносить в partials только если это улучшает поддержку;
- не дробить страницу на слишком много мелких файлов без пользы.

## 12. Роутинг

Роутинг должен решать четыре задачи:

- определить язык;
- определить страницу;
- вернуть 404 для неизвестного route;
- при необходимости выполнить редирект на явный языковой URL.

Рекомендуемый контракт route:

```php
[
    'locale' => 'en',
    'page' => 'home',
    'status' => 200,
    'redirect_to' => null,
    'explicit_locale' => true,
]
```

Правила:

- неизвестная страница должна отдавать корректный `404`;
- редирект с `/` на язык должен быть `302`, чтобы не закреплять автоматический выбор навсегда;
- ручная смена языка может запоминаться cookie;
- все пути страниц должны строиться по `config('pages')`;
- canonical и sitemap должны использовать те же функции URL, что и роутинг.

## 13. SEO-архитектура

SEO должно быть встроено в движок с первого дня.

Обязательно:

- уникальный `title` для каждой страницы и локали;
- уникальный `meta description`;
- корректный `<html lang="">`;
- canonical на текущую языковую версию;
- `hreflang` на все языковые версии, включая self-reference;
- `x-default`;
- Open Graph title/description/image;
- Twitter Card;
- JSON-LD;
- `robots.txt`;
- `sitemap.xml`;
- локализованные `alt`;
- человеко-понятные URL.

Для многоязычного сайта:

- `hreflang` URL должны быть абсолютными;
- canonical не должен всегда указывать на fallback-язык;
- sitemap должен включать только реальные индексируемые URL;
- `/` как auto-redirect не должен конкурировать с языковыми страницами;
- нельзя смешивать языки в одном HTML.

`meta keywords` не является обязательным SEO-фактором и не должен быть частью
критерия готовности.

## 14. Производительность и Core Web Vitals

Цели:

- `LCP <= 2.5s`;
- `INP <= 200ms`;
- `CLS <= 0.1`;
- проверять 75-й перцентиль отдельно для mobile и desktop.

Правила:

- серверный HTML должен приходить быстро;
- не использовать тяжелый JavaScript без необходимости;
- не lazy-load изображение первого экрана, если оно является LCP;
- для hero/LCP-изображения использовать правильный размер и при необходимости
  `fetchpriority="high"`;
- для остальных изображений использовать `loading="lazy"`;
- у изображений должны быть `width`/`height` или `aspect-ratio`;
- для больших изображений использовать responsive variants через `srcset/sizes`;
- включить gzip/brotli на сервере;
- настроить cache-control для ассетов;
- CSS и JS держать компактными;
- критичный контент должен быть доступен без JavaScript.

## 15. Браузеры и доступность

Поддерживаемые браузеры:

- Chrome;
- Safari;
- Firefox;
- Edge;
- iOS Safari;
- Android Chrome.

Минимальные требования:

- валидный HTML5;
- UTF-8;
- `<meta name="viewport">`;
- один основной `h1` на страницу;
- логичная структура заголовков;
- доступные кнопки и ссылки;
- видимые focus-состояния;
- формы с `label`;
- сообщения ошибок рядом с полями;
- alt-тексты для meaningful images;
- `alt=""` для декоративных изображений;
- отсутствие console errors в production.

## 16. Безопасность

Даже простой сайт должен быть безопасным.

Обязательно:

- экранировать вывод в правильном контексте: HTML, attribute, URL, JS;
- не вставлять пользовательские данные напрямую в HTML;
- не использовать пользовательский HTML без sanitization;
- валидировать формы на сервере;
- использовать CSRF для форм;
- добавить rate limit на отправку форм;
- не хранить секреты в репозитории;
- не показывать пользователю stack trace;
- писать технические ошибки в `storage/logs`;
- закрыть внешний доступ к `config`, `includes`, `storage`, `content`, `docs`, `TZ`.

Рекомендуемые security headers:

- `Content-Security-Policy`;
- `X-Content-Type-Options: nosniff`;
- `Referrer-Policy: strict-origin-when-cross-origin`;
- `Permissions-Policy` с минимальными разрешениями;
- `Strict-Transport-Security` после включения стабильного HTTPS.

## 17. Формы и обработчики

Если сайт содержит форму заявки:

- форма должна работать без JavaScript;
- JavaScript может улучшать UX, но не быть единственным способом отправки;
- server-side validation обязательна;
- все ошибки должны быть локализованы;
- success/error-сообщения должны быть локализованы;
- email получателя и SMTP-настройки должны быть в конфиге/секретах;
- отправку нужно логировать без сохранения лишних персональных данных;
- добавить anti-spam: honeypot, CSRF, rate limit.

## 18. Ассеты

Правила ассетов:

- логотипы хранить в `assets/logos/`;
- иконки в `assets/icons/`;
- изображения страниц в `assets/images/`;
- social preview в `assets/social/`;
- не хранить тяжелые исходники в production-пути;
- использовать понятные имена файлов;
- избегать пробелов и кириллицы в именах production-файлов;
- для важных изображений подготовить mobile/desktop или responsive variants;
- social image должен иметь стабильный путь из конфига.

## 19. Документация проекта

Каждый новый сайт должен иметь минимум:

- `docs/00_PREDEVELOPMENT_ANALYSIS_RU.md`;
- `docs/01_DEVELOPMENT_RULES_RU.md`;
- `docs/02_LOCAL_RUN_RU.md`;
- `docs/03_DEVOPS_DEPLOY_RU.md`;
- `TZ/01_SITE_ENGINE_ARCHITECTURE_TZ_RU.md`.

В документации должны быть зафиксированы:

- стек;
- языки;
- fallback-язык;
- URL-модель;
- структура директорий;
- правила локализации;
- SEO-требования;
- правила деплоя;
- release checklist.

## 20. Локальный запуск

Минимальный локальный запуск:

```bash
php -S 127.0.0.1:8080 router.php
```

Для локального запуска нужно проверить:

- `/` редиректит на выбранный язык;
- `/en/`, `/ru/`, `/es/` открываются;
- неизвестная страница отдает `404`;
- sitemap и robots доступны;
- формы не падают без SMTP;
- в PHP error log нет ошибок.

## 21. Деплой

Для простого хостинга:

- загрузить файлы сайта;
- выставить PHP 8.4 или минимум PHP 8.3;
- настроить домен и HTTPS;
- закрыть доступ к внутренним папкам;
- настроить rewrite rules;
- настроить SMTP/секреты;
- проверить права на `storage/logs`;
- проверить `robots.txt` и `sitemap.xml`;
- проверить security headers;
- проверить страницы во всех языках.

Перед релизом нельзя публиковать сайт, если:

- есть PHP warnings/notices;
- language switcher ведет на неверные URL;
- canonical/hreflang конфликтуют;
- форма отправляет пустые заявки;
- есть смешение языков;
- мобильная верстка ломается;
- social preview не настроен;
- внутренние папки открываются из браузера.

## 22. Критерий готовности движка

Движок считается готовым, если новый язык добавляется без копирования страниц:

1. добавить файл `locales/{lang}.php`;
2. добавить язык в `config/app.php`;
3. проверить language switcher;
4. проверить `hreflang`;
5. проверить sitemap;
6. заполнить переводы;
7. пройти визуальную проверку длинных строк.

Если для нового языка нужно копировать все страницы и вручную заменять тексты,
архитектура сделана неправильно.

## 23. Антипаттерны

Нельзя:

- делать отдельный независимый сайт для каждого языка;
- хранить видимые строки прямо в шаблонах;
- переводить сайт через JavaScript после загрузки;
- использовать `?lang=en` как основную URL-модель;
- хранить SMTP-пароли в репозитории;
- публиковать сайт без sitemap;
- публиковать сайт без localized meta;
- использовать один canonical для всех языков;
- оставлять `/config/` или `/storage/` доступными из браузера;
- строить простой сайт на тяжелом frontend-стеке без причины;
- принимать форму только на клиентской валидации;
- считать дизайн готовым без мобильной проверки.

## 24. Стартовый checklist нового сайта

Перед началом нового проекта нужно ответить:

1. Какой бренд и домен?
2. Какие языки v1?
3. Какой fallback-язык?
4. Какие страницы v1?
5. Нужна ли форма?
6. Нужны ли legal pages?
7. Какие SEO-страницы должны индексироваться?
8. Есть ли готовый content source of truth?
9. Какие изображения нужны на первом экране?
10. Какой хостинг и какая версия PHP доступны?

После ответов можно создавать каркас проекта:

```text
config/
includes/
locales/
templates/
handlers/
assets/
static/
storage/
content/
docs/
TZ/
```

## 25. Итоговое правило

Хороший многоязычный PHP-сайт должен быть простым по стеку, но взрослым по
архитектуре. Простота не означает хаос. В хорошем проекте есть понятный
роутинг, локализация, SEO, безопасность, performance, документация и проверка
перед релизом.
