# Правила разработки сайта

Статус: `draft-for-development`

## 1. Цель

Этот документ фиксирует правила разработки для многоязычного сайта
`Core AI Platform`. Его задача - сделать проект не разовой версткой, а
поддерживаемым публичным сайтом с понятной архитектурой, SEO, безопасностью и
контролем качества.

## 2. Общие принципы

- Сайт строится как серьезный B2B-продукт, а не как одноразовый лендинг.
- Контент, шаблоны, конфигурация и служебная логика разделяются.
- Все языки используют один и тот же набор шаблонов.
- Все видимые строки должны проходить через систему локализации.
- SEO-метаданные являются частью локализации.
- Любое новое решение должно быть простым для хостинга и сопровождения.
- Нельзя добавлять технологию только потому, что она модная.

## 3. PHP-правила

- Целевой production-runtime: `PHP 8.4`.
- Минимальный production-runtime: `PHP 8.3`.
- `PHP 8.2` допустим только временно, если хостер еще не поддерживает `8.3/8.4`.
- Не начинать новый production-сайт на неподдерживаемой версии PHP.
- В каждом PHP-файле включать `declare(strict_types=1);`.
- Не смешивать большие блоки бизнес-логики и HTML в одном файле.
- Не писать SQL/базу данных для v1, если нет прямой задачи.
- Все настройки держать в `config/`.
- Все служебные функции держать в `includes/`.
- Все шаблоны держать в `templates/`.
- Все обработчики форм держать в `handlers/`.
- Не хранить секреты в репозитории.
- Для вывода текста в HTML использовать helper экранирования.

## 4. Правила локализации

- Базовая fallback-локаль v1: `en`.
- Поддерживаемые локали v1: `ru`, `en`, `es`.
- Новый язык добавляется через новый файл в `locales/` и запись в `config/app.php`.
- Структура ключей во всех локалях должна быть одинаковой.
- Нельзя оставлять строки напрямую в шаблоне, если они видимы пользователю.
- Переводиться должны меню, кнопки, формы, ошибки, alt, meta и SEO-тексты.
- Если ключ отсутствует в выбранном языке, используется fallback на `en`.
- Языковые URL должны генерироваться централизованно.
- Все публичные языковые версии должны иметь явный префикс: `/en/`, `/ru/`, `/es/`.
- URL без языкового префикса используется только как входная точка для
  автоопределения языка и временного `302`-редиректа.
- Автоопределение языка должно идти в таком порядке: cookie ручного выбора,
  затем `Accept-Language` браузера с учетом `q`, затем fallback на `en`.
- Явный URL языка нельзя перебивать автоопределением: `/ru/platform/` всегда
  остается русской страницей, `/en/platform/` - английской.
- Переключатель языка должен вести на явные URL и запоминать выбранную локаль.

## 5. Правила шаблонов

- Шаблон отвечает за структуру, а не за хранение контента.
- Части страницы выносить в partials только если это реально улучшает поддержку.
- Не дробить страницу на десятки микрофайлов без необходимости.
- Компоненты header, footer, language switcher и head должны быть переиспользуемыми.
- В шаблонах нельзя дублировать одни и те же SEO-теги вручную для разных языков.

## 6. Правила CSS и UI

- CSS должен поддерживать серьезный B2B-образ продукта.
- Не делать сайт визуально похожим на шаблонный крипто/AI-лендинг.
- Не использовать чрезмерные градиенты, декоративные пятна и случайные эффекты.
- Проверять мобильную версию наравне с desktop.
- Длинные английские/испанские строки не должны ломать карточки и кнопки.
- Все интерактивные элементы должны иметь hover/focus-состояния.
- Карточки, сетки и панели должны иметь стабильные размеры и не прыгать от текста.

## 7. Правила JavaScript

- JavaScript используется только для интерактива: меню, модальные окна, формы,
  переключатели, небольшие улучшения интерфейса.
- Основной контент должен быть доступен без JavaScript.
- Нельзя строить мультиязычность через замену текста на клиенте после загрузки.
- Не добавлять тяжелые библиотеки без явной необходимости.
- Ошибки JS не должны блокировать чтение страницы.

## 8. SEO-правила

Для каждой публичной страницы и локали должны быть:

- `title`;
- `meta description`;
- canonical URL;
- `hreflang` для всех языков;
- `x-default`;
- Open Graph title/description/image;
- Twitter Card;
- корректный `html lang`;
- локализованные alt-тексты;
- запись в sitemap.

Для многоязычности:

- каждая языковая версия должна ссылаться через `hreflang` на себя и остальные версии;
- `hreflang` URL должны быть абсолютными;
- `x-default` должен вести на английскую fallback-страницу;
- `canonical` должен указывать на текущую языковую версию, а не всегда на русский URL;
- sitemap должен отражать реальные индексируемые URL.

Запрещено:

- использовать один и тот же `title` для всех страниц;
- оставлять русские SEO-тексты на английской или испанской странице;
- делать языковые версии через query-параметры вида `?lang=en` как основную URL-модель;
- публиковать страницу без meta description;
- считать `meta keywords` обязательным SEO-фактором.

## 9. Правила контента

- `content/` является редакционной зоной и источником смыслов.
- Контентные документы не должны подключаться напрямую как runtime-данные.
- Перед переносом в локали текст должен быть структурирован по ключам.
- Нельзя смешивать черновые формулировки с production-copy без статуса.
- Бренд `Core AI Platform` пока считается рабочим именем и должен быть легко заменяемым.

## 10. Безопасность

- Все данные формы валидировать на сервере.
- Для формы использовать CSRF-токен.
- Добавить rate limit для отправки формы.
- Не выводить пользователю stack trace или PHP warnings.
- Логи писать в `storage/logs`.
- Доступ к `config`, `includes`, `storage`, `content`, `docs`, `TZ` должен быть закрыт снаружи.
- Секреты хранить вне репозитория или в локальном конфиге, который не коммитится.
- Все выводимые строки экранировать в правильном контексте: HTML, attribute, URL, JS.
- Не вставлять переменные напрямую в inline JavaScript.
- Не использовать пользовательский HTML без sanitization.
- Настроить базовые security headers: `Content-Security-Policy`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `Strict-Transport-Security` после HTTPS.

## 11. Производительность и браузеры

- Цель Core Web Vitals: `LCP <= 2.5s`, `INP <= 200ms`, `CLS <= 0.1`.
- Проверять 75-й перцентиль отдельно для mobile и desktop.
- Не lazy-load изображение первого экрана, если оно является LCP.
- Для hero/LCP-изображения использовать правильный размер, preload или `fetchpriority="high"` по ситуации.
- Для остальных изображений использовать lazy loading.
- Указывать `width`/`height` или `aspect-ratio` для изображений и медиа.
- Использовать `srcset`/`sizes` для крупных responsive images.
- Оптимизировать изображения и хранить тяжелые оригиналы вне production-пути.
- Включить gzip/brotli на хостинге.
- Настроить кеширование ассетов с версионированием.
- HTML должен работать без JavaScript для чтения основного контента.
- Проверять Chrome, Safari, Firefox, Edge, iOS Safari и Android Chrome.

## 12. Проверка перед релизом

Перед публикацией нужно проверить:

- главная страница открывается на `ru`, `en`, `es`;
- fallback локализации работает;
- language switcher ведет на правильные URL;
- sitemap содержит все публичные URL;
- robots.txt не блокирует нужные страницы;
- canonical и hreflang корректны;
- нет смешения языков на странице;
- форма не отправляется с пустыми/невалидными данными;
- мобильная верстка не ломается;
- нет ошибок в PHP error log;
- нет ошибок в browser console;
- изображения оптимизированы;
- social preview выглядит нормально;
- security headers проверены;
- страница проходит Lighthouse/PageSpeed без критичных проблем;
- Search Console после публикации видит sitemap и индексируемые страницы.

## 13. Критерий готовности движка

Движок сайта можно считать готовым, когда новый язык добавляется без копирования
страниц:

1. добавить файл `locales/{lang}.php`;
2. добавить язык в `config/app.php`;
3. добавить route/entry для языкового URL;
4. проверить sitemap/hreflang;
5. заполнить переводы.

Если для добавления языка нужно копировать всю страницу и вручную менять тексты,
архитектура считается неправильной.

## 14. Ориентиры качества

При разработке использовать как ориентир:

- актуальные поддерживаемые версии PHP;
- Google Search Central для SEO, `hreflang`, sitemap, structured data;
- web.dev для Core Web Vitals;
- MDN для HTTP headers и браузерной совместимости;
- OWASP для XSS, output encoding и обработки пользовательских данных.
